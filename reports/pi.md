# Raspberry Pi Tier Audit

## Health: 4 — Mature, fail-closed-disciplined Pi tier; the live issues are footprint bloat and an IPv6 brute-force gap, not safety.

## Top findings (max 3, most important first)
1. **PI-001 (P2):** Login lockout is keyed on the exact client IP, but the portal now listens dual-stack on `::`. A LAN attacker holding a typical /64 can rotate IPv6 source addresses to get a fresh failure budget per address, defeating the passcode brute-force throttle. The `is_lan_addr` guard already reasons in /64s; the rate limiter does not.
2. **PI-002 (P2):** Test-only packages (`pytest`, `responses`, `hypothesis`) are floors in `hardware/requirements.txt`, and both `install.sh` and `update.sh` install that file into the service venv on every boot — so the unattended field device carries (and re-resolves on each restart) deps no runtime module imports. Direct hit to the "minimal footprint" mission.
3. **PI-006 (P3):** `git reset --hard origin/main` on every boot/restart pins the Pi to whatever is currently on `main` with no known-good fallback ref. A bad commit on `main` propagates to the field on the next reboot; recovery requires a human + a new commit. Accepted by design, but worth an explicit guard (verify importability before declaring the pull good).

## Cross-domain handoffs
- **PI-001 fix touches `provision/auth.py` (shared service lib).** The `RateLimiter` is shared by the Pi portal AND the cloud/console (`provision`), so a /64-normalization change should be reviewed by whoever owns the control-plane/console tier (cloud) to confirm it does not regress the console's `locked()/record_fail()` facade. The Pi owns the portal call sites; the shared algorithm is a coordination point.
- **CDN-first reads policy is satisfied at the edge, not the Pi.** `portal._archive_image_bytes` (portal.py:528-534) reads stored frames via the edge's `/api/archive/image` endpoint, never directly from FOS; the prune/wipe paths (`usage_stats.fos_prune`/`fos_wipe`, portal.py:682,718) are LIST/DELETE (the documented exception), not reads. Whether `/api/archive/image` itself CDN-signs is an EDGE-tier concern — no Pi finding, flagging only so the edge auditor confirms that endpoint stays CDN-first.
- **No cloud-edge-logs / inference-on-edge policies hold on the Pi side.** Telemetry is Pi-local SQLite only (telemetry.py:38-39, `/var/lib/garden-protector`); the camera/gateway push paths POST images to the edge and explicitly IGNORE the verdict (camera_daemon.py:461-472, camera_pusher.py:14-16) — no inference runs on the Pi. Nothing to hand off; noted for completeness.

# Fastly Edge (Rust→wasm) Audit
## Health: 4 — Safety/auth/CDN/SSOT policies are correctly fail-closed and well-tested; the live gaps are in the deploy mechanics (broken `make deploy-edge`, no purge), not the running code.

## Top findings (max 3, most important first)
1. **`make deploy-edge` cannot ship a code change (EDGE-001).** The `deploy-edge` target runs `build` (which only refreshes `backend/bin/main.wasm`, never the tarball) then `deploy_edge.py`, whose default package is `backend/pkg/garden-protector-backend.tar.gz`. The script's stale-guard (deploy_edge.py:71) then *refuses* because the package is older than the wasm. So `make deploy-edge` errors out on every real code change — the documented `fastly compute pack` step is missing from the Makefile.
2. **Deploy never purges the CDN (EDGE-002).** Neither `deploy_edge.py` nor `make deploy-edge` issues `purge_all`. Static assets are now ETag + `no-cache,s-maxage` (routes.rs:79) which removes the hard-refresh pain, but the standing policy still requires a post-deploy purge + propagation wait; an operator following `make deploy-edge` skips it silently.
3. **Edge code is genuinely fail-closed and the invariants are tested (positive).** `handle_status` defaults to `continue=false` on any `garden_state` read error (main.rs:1324-1340), non-default gardens fail closed (main.rs:1290), auth fails closed on store error (auth.rs:91-114), rain veto can only downgrade (safety.rs:65-81), and 70 native tests pass including `test_heartbeat_continue_fails_closed_on_unreadable_flags`. No P0/P1 correctness defects found in the request path.

## Cross-domain handoffs
- **EDGE-002 (purge) partially overlaps cloud/CI**: the actual `purge_all` could live in the deploy script (edge owns it) but a CI/release pipeline step is the *tests/ci* expert's domain if they own release automation. Edge owns `deploy_edge.py`; coordinate so the purge isn't added in two places.
- **CDN read-signing service (`cdn_read` backend) provisioning** — the edge correctly *consumes* it (archive.rs:321-353, routes.rs:502) but the VCL service that strips `/img/` and SigV4-signs to the private bucket, plus registering the `cdn_read`/`fos_archive`/`twilio_api` backends with correct `override_host`/`ssl_sni_hostname`, is the **cloud** expert's turf (provision/vcl.py). Wrong backend fields cause silent 503s — flag for cloud to verify on every reprovision.
- **`device_config` rate-limit keys + Config Store seeding** (`rate_limit_max_fails`, `rate_limit_window_s`, `is_armed`) are written by **cloud** provisioning; the edge only reads them (main.rs:144-155, 517-520). Any rename is a cross-tier contract change → endpoint-contract.md + spec.toml.
- **dashboard.html / timelapse.html shared JS/CSS behavior** (the `__ASSET_VERSION__` splice, `GP_VIEW_ONLY`, nav rendering) — the edge splices them server-side (routes.rs:105-114, contract_gen render_nav), but the client-side `ui/static/gp.*` logic is the **ui** expert's domain.

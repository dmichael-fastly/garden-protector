# Fastly Cloud Control Plane Audit
## Health: 4 — Solid, policy-aligned control plane; the main real risks are an unpaginated find-or-create and an eventual-consistency window in the "single-writer" registry hydrate.

## Top findings (max 3, most important first)
1. **CLOUD-001 (P1):** `find_service_by_name` reads only the FIRST page of `GET /service`. On an account with more services than the default page size, an existing service is missed and `provision()` creates a DUPLICATE — leaking a fresh FOS bucket + scoped key and a second set of stores. Benign on the small test account today, latent on any busier account.
2. **CLOUD-002 (P1):** The registry's "single writer, no CAS needed" guarantee silently degrades on a host whose local mirror is missing/lost: `_load_mirror` hydrates from EVENTUALLY-CONSISTENT KV reads, so a fresh-host write right after a prior write can clobber gardens/devices (the exact lost-update the design claims to prevent).
3. **CLOUD-003 (P2):** `kv_get` round-trips registry blobs through `fastly()`, which `json.loads` then re-`json.dumps`. This is lossy for byte-exact data and reorders/reformats JSON; it works for registry docs by luck (they're re-parsed) but is a fragile transport for a store that also holds the uploaded ONNX model bytes.

## Cross-domain handoffs
- **CLOUD-005 (EDGE):** The authoritative fix for the stale archive LIST is on the Rust edge — the FOS `ListObjectsV2` send in `backend/src/archive.rs` needs `CacheOverride::pass()` so listings are never cached. The control plane can only mitigate by purging after a wipe/prune. Hand the code change to the edge expert.
- **CLOUD-007 (EDGE/UI):** `default_garden_name` is rendered into the edge dashboard header server-side; the edge must HTML-escape it. The cloud side should still length-bound/sanitize, but output-encoding correctness is an edge/UI concern.
- **CLOUD-005 cache TTL (EDGE):** The CDN read VCL marks images `immutable, max-age=2592000` (vcl.py is cloud-owned), but the consequence — deleted images still 200ing from browser/edge cache after a wipe — is felt on the edge/UI read path; coordinate the purge story with the edge expert.
- **Pi-side bulk delete (PI):** `usage_stats.fos_wipe`/`fos_prune` are documented as running on the Pi (edge can't bulk-delete due to the ~16-backend-send cap). The retention SCHEDULER that invokes them lives in the Pi portal — scheduling cadence/retention-floor UX belongs to the Pi expert; the cost-floor math (30-day min) is correctly modeled here in `cost_rates.estimate_capture_monthly`.

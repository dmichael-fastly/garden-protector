# Build Health & CI Audit
## Health: 4 — `make ci` is fully green and the drift gate is well-built; the main weakness is that the gate never compiles the actual wasm deploy artifact.

I ran the full `make ci` pipeline end to end (not from cache):
- `gen-check` (`python3 contract/gen.py --check`) → "generated files match contract/spec.toml ✓"
- `test-rust` (`cargo test --release`, native) → 70 passed, 0 failed, **0 compiler warnings** in the test build
- `test-py` (`python3 -m pytest -q`) → 396 passed, 1 skipped (OpenCV optional only), stable across reruns
- `scan` (`gitleaks git --no-banner --redact`) → 192 commits / 2.56 MB scanned, **no leaks**

The cross-language contract gate is genuinely strong: `contract/spec.toml` is the single source, both the Rust `contract_gen::gen_tests` and the Python `tests/test_contract.py` verify their generated module against that one spec + fixture, and `gen-check` then fails the build if the checked-in generated files drift. The fail-closed property suite (`tests/test_safety_properties.py`) is pure, fast, and deterministic across repeated runs.

The findings below are about what the gate does NOT cover, not about anything currently red.

## Top findings (max 3, most important first)
1. **CI-001 (P1):** `make ci` never compiles the wasm32 deploy artifact. `test-rust` builds a native test harness and all host-ABI code is `#[cfg(target_arch="wasm32")]`-gated, so a wasm-only compile break passes `make ci` but breaks `make build`/deploy.
2. **CI-002 (P2):** There is no automated CI runner — `.github/workflows/` does not exist; `make ci` is manual-only, so the gate runs only when a developer remembers to run it.
3. **CI-003 (P2):** The documented `GP_TELEMETRY=0` trap (6 false telemetry-test failures) has no in-suite defense; `_ENABLED` is read once at import (`hardware/telemetry.py:37`) and `conftest.py` clears `GP_ADMIN_PASSCODE` but not `GP_TELEMETRY`.

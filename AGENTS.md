# AGENTS.md - Fastly Demo Engineering Guide

## User Persona

You are a developer with extensive experience building web based applications and APIs. You are an expert in Fastly and using Fastly products like VCL and Compute to build world-class, scalable and secure systems at the edge.

## Mission & Objective

Your primary objective is to rapidly build, deploy, and document functional examples of how to use Fastly effectively. 

The core KPI is to ship **one new, functional Fastly demo or use-case implementation per day**.

Every demo must simultaneously teach two concepts:
1. The generic industry problem being solved, using terms a non-Fastly searcher would use.
2. The Fastly-native implementation, using accurate Fastly product names, commands, and workflows.

* **High Autonomy:** Pick a practical path and ship a working demo rather than waiting for ideal requirements.
* **Organic Diversity (CRITICAL):** Do not make these repositories look like uniform corporate sales pitches. Vary your documentation style, file organization (where practical), and tone. Some repos should be highly technical and terse; others can be more narrative or tutorial-based. **(⛔ DOES NOT APPLY to garden-protector — see [CHARTER.md](CHARTER.md). This repo is one product and standardizes deliberately: one palette, one auth library, one router, one contract.)**
* **Scrappy but Complete:** Prefer a small complete example over a large incomplete framework. 
* **Safety First:** Use public-safe sample data only. Never include customer names, internal account details, real credentials, or private architecture claims. Fail closed for security controls.
* **Factual Accuracy:** Treat official Fastly documentation as the source of truth for limits and features. Do not invent pricing or performance claims.

## Definition of Done

A demo is not complete until all of the following are true:

* **Runnable Code:** Contains a working Fastly implementation (VCL, Compute, Terraform, Fastly CLI, API).
* **Documentation:** Includes a clear `README.md` using both Fastly and generic terminology.
* **Validation:** Includes at least one copy-pasteable smoke test using `curl` or a documented flow.
* **No Secrets:** Never commit real secrets, tokens, or account IDs. If the demo reads from runtime environment variables or similar configuration, include a `.env.example` file. Demos that use Fastly Secret Store / Config Store for credentials and configuration don't need a `.env.example` — document the required keys (and how to seed them via `fastly secret-store-entry create` / `fastly config-store-entry create`) in the README instead.
* **Lifecycle Paths:** Documents a local path (where possible), a deployed path, and a teardown/cleanup section.

## Required README.md Elements (Organic Presentation)

Do not use a single, rigid template for your `README.md` files. To ensure the repositories look organic and diverse, **shuffle the order, vary the heading names, and change the formatting style** from project to project.

However, every `README.md` **must contain** the following core components in some form:

* **Problem Statement:** Explain the problem in generic terms (e.g., API rate limiting, bot mitigation, LLM semantic caching).
* **The Fastly Solution:** Explain the Fastly-specific fit (e.g., global edge network, Next-Gen WAF, KV Store).
* **Architecture / Request Flow:** Include a text diagram or explain the flow between the client, edge, services, and origin. Map Fastly products to the generic capabilities they provide.
* **Prerequisites:** List account requirements, CLI/tooling, language runtimes, and origin requirements.
* **Deployment Instructions:** Step-by-step deployment using Fastly CLI, Terraform, or the Fastly Agent Toolkit. Include the fastest local or simulated path if applicable.
* **Validation / Smoke Test:** Commands that prove the behavior works, such as `curl` examples with expected headers or status codes.
* **Production Considerations:** Discuss secrets, origin protection, TLS, cache keys, or limits.
* **Teardown / Cleanup:** Explain how to remove services, stores, and resources to prevent ongoing billing.

## Known Traps

* **`config-store-entry create` / `secret-store-entry create` take `--store-id`, not `--name`.** The parent `create` command identifies stores by name; the entry-create commands identify them by ID. Capture the ID at create time with `--json | jq -r '.id'` (or look it up later with `fastly <resource> list --json`). The same applies if you reuse an existing store created via the UI.
* **`secret-store-entry create` does not have a `--secret=` flag.** The secret *name* is `--name`; the secret *value* comes from `--stdin`, `--file`, or an interactive prompt. Piping the value into `--stdin` is the right pattern for scripts.
* **Local-vs-cloud detection: verify the runtime hostname equals `localhost`, not whether `FASTLY_SERVICE_ID` is empty.** Viceroy populates `FASTLY_SERVICE_ID` with a non-empty placeholder, so the obvious "is the ID empty?" check fails closed in cloud and open locally — the opposite of what you want. The hostname is `localhost` under Viceroy and the POP hostname in production.
* **Build System Configurations:** Ensure your compiler or linter config files use extensions and module types that align with your build tools to avoid module-resolution or compile-time warnings (e.g., ESM vs. CommonJS issues in JS, or Cargo configurations in Rust).
* **Regex escaping inside character classes:** Avoid unnecessary escapes of range symbols (e.g., `[+-]` instead of `[+\-]`). Linters and engines often flag unnecessary escapes inside character classes where the character's position already makes its meaning literal.
* **Inline event handlers in frontend markup:** Bind frontend event handlers programmatically in scripts instead of inside inline markup (e.g., avoid inline attributes like `onclick`). This ensures static analysis and linter rules can correctly trace variable and function usage.
* **Three-state theme toggle (auto / light / dark) needs a `data-theme` attribute on `<html>`, not a class.** `:root[data-theme="light"]` has higher specificity than `:root` inside `@media (prefers-color-scheme: light)`, so an explicit choice always wins. Auto = no attribute set, which lets the media query take over. The light palette has to be declared twice (once in `@media (prefers-color-scheme: light) { :root:not([data-theme="dark"]) { … } }`, once in `:root[data-theme="light"] { … }`) because CSS can't share a declaration block across the @media boundary. Also remember `color-scheme: dark|light` so native scrollbars/form chrome match.
* **Local dev server hot-reload limits:** Static files or binary assets baked/compiled directly into the WebAssembly binary at build time are not picked up by the local simulation file watcher. You must restart the dev server after editing those compiled assets. Standard source script changes hot-reload normally.
* **Two `position: fixed` elements in the same corner will overlap silently.** If you stack a status badge and a theme toggle in the top-right, give the inner element (`#status`) explicit room: `right: <toggle_width + offset + gap>px`. They don't participate in normal layout, so flex/grid won't save you — pick the offsets by hand.
* **`time 0.3.x` compilation failures on newer Rust compilers:** Transitive dependencies on older versions of the `time` crate (e.g. `0.3.23`) will fail to compile on modern Rust compilers due to type inference errors. Fix this instantly by adding an explicit, newer version dependency (e.g. `time = "0.3.36"`) to your `Cargo.toml` dependencies block.
* **Fastly Rust SDK v0.9.x API differences:**
  - `ConfigStore::open("name")` is infallible and returns `ConfigStore` directly, not a `Result` or `Option` (do not use the `?` operator).
  - `KVStore` is named `KVStore` (not `KeyValueStore`), and `store.lookup("key")?` already executes the retrieval returning `Result<Option<Body>, Error>` directly (do not chain `.execute()`).
  - Serializing json payloads on responses requires `.with_body_json(&payload)` instead of `.with_json(&payload)`.
* **Rust toolchain target resolution and WASI Preview 1 renaming:** The official WASI target name is `wasm32-wasip1`. If Homebrew standalone compilers override your environment path and throw "can't find crate for core" errors, explicitly force `rustup` shims by exporting `PATH="$HOME/.cargo/bin:$PATH"` at the start of any build and execution scripts. **(Phase 0 added `backend/rust-toolchain.toml`, which declares the `wasm32-wasip1` target so rustup auto-installs it on a fresh checkout. BUT only the rustup shim `~/.cargo/bin/cargo` honors that file — Homebrew's cargo ignores it — so the `PATH="$HOME/.cargo/bin:$PATH"` prefix in the Makefile/`serve_backend.sh` is STILL required to select the rustup cargo. The toolchain file makes that cargo reproducible; it does not reorder PATH.)**
* **Client-Backend MIME Form vs Raw Body Mismatch:** When the Fastly Compute Rust backend decodes incoming images via raw body bytes, ensure the client POSTs raw binary bytes directly (`data=image_bytes` and `Content-Type: image/jpeg`) rather than `files=...` which wraps the payload in `multipart/form-data` and injects boundary strings that break pure JPEG decoders.
* **Viceroy Local KV Stores Configuration:** For local Viceroy simulation, `[local_server.kv_stores]` must define keys/files using inline tables (e.g. `[[local_server.kv_stores.<store_name>]]` with `key = "name"` and `file = "path"`) instead of folder paths, which can trigger parsing errors inside the Fastly CLI.
* **Dynamic Server Polling in E2E Tests:** When starting `fastly compute serve` in background bash processes for automated testing, use curl-based loop polling to verify when the local server is listening on the target port before running client triggers, rather than hardcoded `sleep` delays. This robustly accounts for compilation times.
* **Tract ONNX Loading Optimization:** Always specify a fixed input shape using `.with_input_fact(...)` (e.g. `f32::fact(&[1, 3, 224, 224]).into()`) when loading ONNX models in Tract. This allows Tract to optimize the graph instantly in <100ms instead of performing dynamic shape analysis which takes several seconds per request in WebAssembly.
* **Tract 0.20 cannot load the MobileNet V2 ONNX (`Failed analyse for node Gemm_104`).** The torchvision export fails graph analysis on `tract-onnx = 0.20.7` even with `.with_input_fact(...)` set, so `load_model_from_kv()` returns `Err` and `MODEL` becomes `None`. Fix: use `tract-onnx = "0.23"`. Note 0.23 needs **rustc ≥ 1.91**, and its API changed — the runnable type is `Arc<TypedRunnableModel>` (no generic arg; `SimplePlan` left the prelude), and tensor data is read via `results[0].to_plain_array_view::<f32>()?` (not `to_array_view`).
* **A "demo fallback" that returns a plausible positive will mask real failures.** The old `/api/evidence` fallback returned `("raccoon", 0.95, "mitigate")` for any JPEG (`image_bytes.contains(&0xff)`, true for every JPEG) whenever the model failed to load. Because the model never loaded (trap above), the whole demo ran on this fallback and *looked* like working ML. Fail **closed/loud** instead: return HTTP 503 when `MODEL` is `None` so a misconfigured KV Store / bad model surfaces immediately.
* **MobileNet V2 ONNX emits raw logits, not probabilities.** The graph ends in `Gemm` with no `Softmax` op. Apply a numerically-stable softmax (subtract max logit before `exp`) before thresholding; a raw-logit `> 0.6` check is meaningless and fires for almost any image. Also gate on a curated critter-class allowlist — stock ImageNet-1k has **no `raccoon` synset**, so true raccoon detection needs a fine-tuned model.
* **Fastly CLI v13 refuses to build an "unvalidated" rustc** (e.g. 1.96): `ERROR: version '1.96.0' of Rust has not been validated for use with Fastly Compute`. Either `fastly update` to CLI ≥ 15, or bypass the CLI's build gate locally: `cargo build --release --target wasm32-wasip1` then run the wasm directly with `viceroy -C fastly.toml --addr 0.0.0.0:7878 bin/main.wasm`. Bind `0.0.0.0` (not `127.0.0.1`) when a remote Pi must reach the dev server.
* **Writable KV Store for mutable edge state (Config Store is read-only at the edge).** The admin dashboard's arm/override flags + the latest event/snapshot live in a **writable** `garden_state` KV Store via `KVStore::insert(&mut self, key, value: impl Into<Body>)` (fastly 0.9.5). Declare it for local Viceroy as seeded array-of-tables (`[[local_server.kv_stores.garden_state]]` with `key`/`data`); it is writable at runtime. **Viceroy keeps inserts in memory for the serve session only — they do NOT persist across `serve` restarts.** Production needs `fastly kv-store create garden_state` + a service resource link (the local `fastly.toml` block does not provide that).
* **`include_str!("dashboard.html")` is a build-time dependency.** `backend/src/dashboard.html` is baked into the Wasm binary; if it is missing/untracked at build time the **whole backend fails to compile** — including the Pi-facing `/api/evidence` and `/api/status`. Commit it alongside `main.rs`. (Per the hot-reload trap above, editing it does not hot-reload — restart the dev server.)
* **Fail-closed applies to the heartbeat, not just the model.** `GET /api/status` returns `continue_mitigation`, the signal that KEEPS deterrents energized (the Pi only stands down on `false` or an HTTP error). If `garden_state` cannot be read, `handle_status` returns **`false`** (unconfirmed `armed` → disarmed, unconfirmed `override` → STOP). Do **not** "simplify" it back to the lenient `read_flags()` the dashboard *display* uses (that defaults ARMED). Only a true open/read **error** fails closed; `Ok(None)` (store unprovisioned) still falls back to the read-only Config Store. Best-effort dashboard writes in `handle_evidence` must likewise never alter the Pi-facing response or the 503 path.
* **Fastly Secret Store names forbid `/`** (only `[A-Za-z0-9._-]`). The RFC's literal `g/<gid>/token_current` is an ILLEGAL secret name. Per-garden tokens use the slash-free encoding `g.<gid>.token_current` / `g.<gid>.token_previous` via ONE shared helper — `token_secret_name` in both `backend/src/main.rs` and `provision/ids.py` (a `pytest` parity test asserts they agree). KV Store keys (e.g. `index/gardens`, `g/<gid>/dev/<did>/…`) DO allow `/` — only the Secret Store is restricted.
* **Fastly Secret Store updates propagate to the edge with a ~15-20 s lag.** A freshly-minted/rotated token is NOT immediately usable: the edge keeps reading the prior value for ~tens of seconds (confirmed live: t=10 s old, t=20 s new). Don't test a just-minted token immediately (you'll get a spurious 401). `gp-provision register-device` mints the token; allow a propagation window before the Pi authenticates with it. (KV is likewise eventually consistent — see the registry trap below.)
* **`ConfigStore::open`/`get` and `SecretStore::get` / `Secret::plaintext` PANIC on a host error** (e.g. an unprovisioned/unlinked store). A cloud deploy that omits `device_config` made `config_flags()` panic → **500 on the heartbeat**. Use the fallible `try_open`/`try_get` on ANY path that may touch a missing store and degrade to safe defaults. (`Secret::plaintext()` has no fallible variant in 0.9.5; the residual panic on a resolved handle still fails SAFE — request aborts → Pi disarms — but reach for `try_get` for the lookup.)
* **Fastly KV is EVENTUALLY consistent — never read-modify-write the registry against KV.** Reading `index/g/<gid>/devices` right after a write may return a stale blob, so successive `register-device` calls lost all but the last device. `gp-provision` keeps an authoritative LOCAL mirror (`configs/{sid}-registry.json`), mutates that, and writes THROUGH to KV (it is the single writer — RFC §4). The edge only READS the registry.
* **Provisioning version flow: clone the active (LOCKED) version before mutating, and a Compute service has no domain until you add one.** Mutating an active version → HTTP 422 `Version locked`; a redeploy must `clone` → mutate the draft → activate. A freshly-created Compute service is unreachable until you `POST …/version/{v}/domain` (e.g. `<name>.edgecompute.app`). Auto-rollback must only delete a service it CREATED this run (a failed *redeploy* of a reused service must not wipe it).
* **Per-garden auth boundary (Step 3):** enforcement is on `/api/control` + the `/api/gardens/*` admin routes AND once-per-trip on `/api/evidence` (a sanctioned bend — 30 s Pi budget, non-default gardens only, default adds ZERO lookups). `/api/status` stays STRICTLY secret-lookup-free (the 3 s heartbeat). The token is looked up under the *claimed* `garden_id`, so a token for garden A is rejected against garden B (the real tenancy boundary). `default` is never issued a token. Admin Response-level behaviors (id→400, auth→401, 405) construct a `fastly::Response`, whose host ABI symbol is absent in the native `cargo test` link — verify those via Viceroy curl, not unit tests.
* **Telemetry must never touch the safety path's latency.** It is tempting to instrument *every* edge KV op with a wrapper that also writes a telemetry ring entry — but `handle_status` opens `garden_state` every ~3 s during mitigation, and `handle_evidence` is the trip→deterrent path the Pi times (Step 8). A synchronous KV ring insert there ADDS latency even though it's "best-effort" (best-effort protects fail-OPEN safety, not latency). Rule: on `/api/status` and `/api/evidence`, telemetry emits to the **log endpoint/stdout only** (batched off-request); the `garden_telemetry` KV ring is written only on non-safety routes (`/api/state`, `/api/snapshot`, `/api/control`, `/api/telemetry`) and the once-per-request summary emitted *after* the response is built in `main()`. On the Pi, telemetry is an enqueue onto a bounded queue drained by a daemon thread — never a synchronous disk/network write on the deterrent/heartbeat path.
* **READS go via the CDN, NEVER direct from FOS (standing rule).** Pull archived/stored files (evidence JPEGs, any object) through the CDN read-signing service (`provision/vcl.py`, public `/img/<key>`), not straight from Object Storage — the CDN caches (cost + latency), keeps the bucket private (it SigV4-signs), and is the intended public read surface; direct FOS reads bypass the cache and add Class-B ops + egress per fetch. **Image reads are aligned:** `handle_archive_image` (`/api/archive/image`) now pulls via the CDN (`cdn_signed_get` over the provisioned `cdn_read` backend), authenticating SERVER-SIDE with the shared secret as the `x-fastly-key` HEADER (never `?key=` in the URL/logs; attached ONLY on the CDN hop — the FOS fallback SigV4-signs and must never carry it), and the edge keeps PROXYING the bytes so the per-garden viewer gate is preserved and the browser never sees the secret. (A browser *redirect* to the CDN would leak the single shared `cdn_secret` to every viewer → cross-garden read — rejected.) It falls back to a direct `fos_signed_get` when the CDN isn't wired (no `cdn_host`/`fos.cdn_secret`) or returns non-200, so un-reprovisioned gardens keep working. Provisioning wires it (`orchestrator.provision`: `cdn_host` in `fos_config`, `fos.cdn_secret` in `garden_tokens`, the `cdn_read` backend) behind an SSRF guard (`fastly_api.safe_cdn_host` — https Fastly hosts only, https Fastly hosts only). **LIST stays FOS-direct BY DESIGN:** `handle_archive_day`/`handle_archive_days` are S3 LIST ops and the CDN VCL deliberately strips LIST params (cross-tenant key-enumeration guard) — a LIST isn't a file pull, so it is NOT a violation of this rule. For any NEW read path, default to the CDN `/img/<key>`. (The best-effort evidence PUT in `handle_evidence` stays direct to FOS — this rule is about READS.)
* **CDN read-signing VCL (the FOS-fronting service) — traps found live-verifying the Step-3 evidence archive (2026-06-21).** `provision/vcl.py` generates the FOS-fronting VCL with two GP-specific changes (a `/img/` prefix strip + a RESTRICTIVE no-LIST query allow-list). The first end-to-end live deploy surfaced a chain of latent bugs:
  - **The public read URL `/img/<key>` MUST be stripped to the bucket-relative key before signing.** The edge writes archive keys as `g/<gid>/<did>/evidence/...` (NO `img/`). The VCL prepends the bucket only when the first path segment isn't the bucket, so a literal `/img/<key>` signs `/<bucket>/img/<key>` -> 404 NoSuchKey. `vcl_recv` strips `^/img/` first.
  - **Gate the shared-secret read on the EDGE only, never the shield hop — via a baked-in `X-Edge-CDN-Auth` secret marker, NOT `Fastly-FF`.** The edge strips `?key=` during cache-key hardening before forwarding to the shield, so re-checking on the shield 403s every shielded request. `Fastly-FF` is a client-spoofable header (would let anyone bypass the gate) — use a `secrets.token_hex(32)` marker the edge stamps in `miss_pass` and the shield matches in `vcl_recv` (strip any client-supplied value first).   - **Set the FOS upstream Host via the backend `override_host` (native Fastly), but SIGN the matching host VALUE in VCL.** Shielding a FOS origin whose fetch carries the CDN's own domain re-enters the same service -> 503 "Same machine same service". `override_host = <region>.object.fastlystorage.app` fixes the sent Host — BUT it is applied AFTER `miss_pass`, so `bereq.http.host` is still the inbound CDN domain at signing time. Sign the computed `var.fosHost` value (NOT `bereq.http.host`) or FOS returns `SignatureDoesNotMatch`. Don't `set bereq.http.host` in VCL; let `override_host` set it and just sign the same value.
  - **`#FASTLY <name>` macros must be on their OWN line** under `sub vcl_X {` (inline `sub vcl_x { #FASTLY x` -> "macro missing" warnings). **`return(purge)` is not a valid action** on this service (omit FASTLYPURGE handling). **`ratecounter`/`penaltybox` must be declared** at top level before use or activation hard-fails. Always `GET /service/{sid}/version/{v}/validate` on a DRAFT before activating.
* **`gp-provision` archive standup idempotency (reused service) — same session.**
  - **`add_backend("fos_archive")` 409s on a reprovision** (backends are inherited when a version is cloned) -> tolerate the duplicate, like `add_domain`.
  - **The FOS access-keys API identifies a key by `access_key` (no `id`/`access_key_id`) and returns `secret_key` ONLY at POST-create.** Delete the temp admin key by `access_key` or it leaks a broad `read-write-admin` key every run. A failed archive run leaves a scoped `gp-<sid>-objects` key whose secret is lost — delete it before retrying so a fresh one is minted.
  - **`ensure_cdn_service` always CREATES a new VCL service** (not find-or-create) -> a failed run leaks an orphaned CDN service; clean it before retry.
  - **Config Store account cap (default 10)** can block `fos_config` creation (`Exceeding max_config_stores`) — raise the limit or free a slot (quota changes take a few minutes to reach the API).
* **POLICY: NO cloud edge logs. Telemetry is Pi-local (SQLite) only.** We deliberately do NOT provision the S3->FOS "Garden Protector Telemetry" logging endpoint anymore — it shipped a gzipped log object roughly once per second (tens of thousands of tiny `telemetry/%Y-%m-%d/%H/…log.gz` objects/day, each a billable Class A write) and dominated the images bucket. `ensure_logging_endpoint` was removed; `provision/fastly_api.py` instead has `delete_logging_endpoint`, and `scripts/remove_cloud_logging.py` strips the endpoint + purges existing objects from already-provisioned services. **To restore edge-log visibility, do it locally/ephemerally — not by re-adding a cloud sink.**
  - Background on how Compute logging works (still true): `println!`/`eprintln!` reach only Fastly's real-time managed stream (`fastly log-tail`), NOT a configured S3/logging endpoint; a configured endpoint only receives what `log_evt` emits to it via `fastly::log::Endpoint::from_name("Garden Protector Telemetry")`. With the endpoint gone, that `log_evt` write is an inert no-op (lines are silently dropped) — harmless, so the edge Wasm did NOT need a rebuild to comply. **Follow-up:** drop the dead `Endpoint::from_name` write from `log_evt` on the next edge deploy. The `#[cfg(target_arch = "wasm32")]` guard on `Endpoint::from_name` must stay until then (native `cargo test` link fails otherwise: `symbol(s) not found ... try_from_name`).
  - **CAVEAT (observed live):** `fastly log-tail` surfaced nothing for this Compute service, so removing the cloud endpoint means the admin console's durable edge-log viewer goes dark. That's an accepted trade for keeping logs out of the cloud; Pi-local telemetry (`hardware/telemetry.py`, SQLite) is unaffected.


* **Admin console / SSE (Step 4) — traps found building the control-plane console (2026-06-21).**
  - **Long-lived SSE belongs in the LOCAL console, not the edge.** Fastly Compute tunes a ~5 s handler budget, so a minutes-long event stream can't live on the edge. "Configure & deploy" is a control-plane activity anyway → the local console (`provision/console.py`, stdlib http.server) streams CLI subprocess stdout as SSE; the edge dashboard keeps polling. Don't try to add an edge `/api/events` SSE.
  - **`gp-provision` prompts for a token even in `FASTLY_MOCK_MODE`.** `cli._resolve_token` requires a token before the (mocked) wire is reached, so a subprocess with stdin closed aborts on the getpass prompt. The console injects a placeholder `FASTLY_API_KEY=MOCK-TOKEN` into the subprocess env in mock mode. Always run provisioning subprocesses with `stdin=DEVNULL` so a missing token fails fast instead of hanging.

* **Process-local statics (`LOCKOUT_STATE`) and WebAssembly isolation:** In standard Fastly Compute execution and local Viceroy simulation, each request is isolated in a fresh WebAssembly instance by default. Process-local `static` or global variables are re-initialized for every request. State-preservation across requests can be achieved either by enabling Fastly's opt-in "Reusable Sandboxes" feature or via external durable databases. Native `cargo test` unit tests should be used to verify the stateful lockout logic in isolated memory, while verifying response status codes via local Viceroy simulation.
* **ConfigStore and host-ABI imports block native cargo test link:** When compiling for native macOS/Linux in unit tests, the Fastly SDK doesn't have native implementations for WebAssembly host imports. Calling `ConfigStore::try_open` or `ConfigStoreHandle::get` directly inside code paths executed during unit tests will fail to compile/link with error `"Undefined symbols ... _get"`. Guard all such host-ABI lookups with `#[cfg(target_arch = "wasm32")]` so they are excluded during native `cargo test` runs, falling back to environment variable overrides or default values on host.

## Testing

**Run `make ci` after every code change.** Fix all errors and warnings. Never report success without running CI. `make ci` = contract drift (`gen-check`) + CSS drift (`css-check`, the committed `ui/static/app.css` vs a fresh Tailwind/DaisyUI build — needs Node) + wasm32 compile check + Rust `cargo test --release` + Python `pytest` + `make scan` (gitleaks). Toolchains are pinned (`backend/rust-toolchain.toml` declares the channel + `wasm32-wasip1` target; `constraints.txt` pins the Python deps the `>=` floors in the `requirements.txt` files install against — see [CHARTER.md](CHARTER.md)).
**`make smoke` is the MANUAL integration gate** for real edge↔Pi end-to-end flows. It runs `tests/smoke_test.sh`: starts `fastly compute serve` locally (~90 s compile), waits for the dev server, then drives `hardware/client.py --trigger` through the stack. Prerequisites: `fastly` CLI installed, live network, `wasm32-wasip1` toolchain. **Do NOT add it to `make ci`** — it requires network + the Fastly CLI and is not suitable for automated or offline runs. Run it manually before any change that touches the edge↔Pi safety path.
**The fail-closed safety suite is named and mutation-checked.** `tests/test_safety_properties.py` carries the cross-tier SAFETY INVARIANT INDEX (heartbeat, Pi-client stop, rain veto, gateway spray core); each invariant has a test that demonstrably fails if broken. Do not weaken these; per CHARTER, a safety-path refactor pins current behavior first.
**Add tests for every non-trivial change.** New endpoint → router test. New utility → unit test. Bug fix → regression test that would have caught it.
**Prefer integration tests over pure mocks** for backend behavior.

## Monitoring and Logging for Auditing

**Telemetry is mandatory and observe-only (standing standard).** Every operation flows
through one control/instrumentation layer that records the action AND the function/route
that triggered it. Concretely:
- **Edge (Rust):** no code calls `KVStore`/`ConfigStore`/`model.run` directly — all host
  I/O and inference go through the `kv_*`/`cfg_*`/`infer` wrappers with a `&TraceContext`.
- **Pi (Python):** every function that does I/O (camera/GPIO/subprocess/HTTP/file) or
  advances the FSM is `@traced` or wrapped in `trace_span`; outbound requests carry
  `X-Garden-Trace-Id`.
- Telemetry is **best-effort, non-blocking, and re-raises the original error unchanged**.
  It may NEVER add a synchronous KV write / network round-trip to, or throw into, the
  trip→deterrent path, the heartbeat, `/api/evidence`, or `/api/status` (see the trap
  below). A PR that adds an un-instrumented I/O path is incomplete.

Scalability (many gardens/devices): **Design for many, ship for one** — carry `garden_id`/`device_id`/`node_id` (default `"default"`) through every new key/event/header now so single→multi needs no migration.

## Traps & Gotchas

* **`gpiozero.is_active` is a `@property` in Python.** When mocking or implementing custom output/deterrent wrappers (like inside `hardware/client.py`), make sure `is_active` is implemented as an `@property` (so it is accessed as `is_active` instead of calling `is_active()`). This maintains seamless compatibility with physical `gpiozero` pins and keeps the test suite green.

## Secrets and Sensitive Data

**Scan for committed secrets BEFORE every commit.** This is wired up: `make ci` runs `make scan` (`gitleaks git` over history; config in `.gitleaks.toml`), and the tracked pre-commit hook `.githooks/pre-commit` runs `gitleaks git --staged` on each commit. Activate the hook once per checkout with `make hooks` (sets `git config core.hooksPath .githooks`). gitleaks must be installed (`brew install gitleaks`); the gate fails closed if it is absent. The live secrets under `configs/` are gitignored, so `gitleaks git` (tracked content only) never sees them. CI also runs security checks and will fail the build if secrets are detected, but catching it locally is faster.
**Never commit a real credential to suppress the scanner.** The point of the gate is exactly this. If a legitimate secret needs to live in the tree (e.g. an SSH public key used as a trust anchor), document why in a comment adjacent to the allowlist entry and explain why exposure is intentional.
**Never put real customer values in code, scripts, tests, or docs.**
